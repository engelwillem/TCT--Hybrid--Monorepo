# VerseHub Domain Audit

## Date: 2026-03-17

## Scope
Auditing the data contracts and page logic inside `src/features/versehub/pages/VersehubReaderPage.tsx` against the backend APIs (`App\Http\Controllers\VerseHubReaderController@getChapterContentApi`).

## Findings
1. **Books Retrieval**: Returns `chapters` map and `books` correctly mapping to `{code, label, testament}`. `VersehubReaderPage` implements this cleanly.
2. **Chapter Content Retrieval**: Backend sends an elaborate payload:
```json
{
  "selected_book": "string",
  "selected_chapter": "integer",
  "chapters": ["integer"],
  "chapter_label": "string",
  "verses": [{"verse": "integer", "text": "string", "href": "string", "key": "string"}],
  "reflection_question": "string",
  "has_reflected": "boolean"
}
```
3. **Frontend Mismatch**: The Next.js frontend is reading `verses`, `chapter_label`, and `chapters`, but it **completely ignores** `reflection_question` and `has_reflected`.
   Instead, the `VersehubReaderPage.tsx` defines:
   ```typescript
   const activeReflectionQuestion = ''; // Placeholder for dynamic logic
   const reflection_question = 'Bagaimana ayat-ayat ini menguatkan imanmu hari ini?';
   ```
   and hardcodes it into `<EndOfChapterPrompt>`. This means the dynamic reflections generated by `VerseHubMentorService` on the backend are currently dead-ending at the UI layer.

## Selected Mismatch
**Hardcoded Chapter Reflection Questions**
The frontend must capture `data.reflection_question` and `data.has_reflected` from the backend JSON payload and pass it to the `EndOfChapterPrompt` to restore full feature parity with the backend mentor engine.
